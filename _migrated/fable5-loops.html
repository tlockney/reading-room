§ 01Two tips for a Mythos-class model

Mythos-class models like Claude Fable 5 have changed the way many of us work at Anthropic. There are two tips worth sharing for getting the most out of this class of models — and both are about building the right loop around the model rather than steering the model directly.

The first is letting the model self-correct against a goal or rubric. The second is letting it manage its own memory across sessions. Each is a kind of loop: one tight (within a task), one wide (across sessions).

§ 02Self-correction loops

There's been a lot of interest in loops recently — Boris Cherny has said that "(his) job is to write loops." Letting a model hillclimb on an evaluation is a common recipe for improving task performance. /goal in Claude Code and Outcomes in Claude Managed Agents are primitives that apply this general recipe to your specific task.

As the prompting guide notes, Fable 5 is good at self-correcting in a loop. A well-designed goal or rubric adds feedback to the environment Claude is running in: Claude runs, collects feedback via the goal or rubric, self-corrects, and proceeds until the criteria are satisfied.

What does the judging matters

Models have trouble with self-critique on their own outputs. A verifier sub-agent tends to outperform self-critique with Fable 5, because grading happens in an independent context window. Outcomes in CMA handles this by spawning a grader sub-agent for you. (More on this failure mode in the harness-design writeup.)

§ 03Parameter Golf — a worked example

Parameter Golf is an open-source ML-engineering challenge: train the best model that fits in a 16MB artifact in under 10 minutes on 8×H100s. It's a bit like Karpathy's autoresearch — it tests an agent's ability to edit basic training code (a single train_gpt.py), launch training, poll the log, read the score, and decide what experiment to run next.

I compared Fable 5 to Opus 4.7 using Claude Managed Agents (CMA), which provides the agent harness plus a hosted sandbox — well-suited to long-running tasks — with 8×H100 GPUs as a self-hosted sandbox. Each test supplied a rubric file with nine checkable criteria (run a baseline, run 20 experiments, and so on), and Parameter Golf ran for up to 8 hours. The Outcomes grader confirmed all criteria were met before letting Claude stop.

Fable 5 improved the training pipeline ~6× more than Opus 4.7 — betting on larger structural changes and showing resilience, pushing through a quantization regression to its biggest win.

Where Fable 5 made structural bets (architecture changes), Opus 4.7's first experiment produced a small win and nearly everything after followed the same template: adjust a scalar, measure, keep if positive.

§ 04Memory as the outer loop

Memory is another area where Fable excels. Think of it as an outer loop spanning sessions: Claude writes to memory during a session, and those memories are retrieved in future ones.

To test it, I used one task from Parth Asawa's team's Continual Learning Bench 1.0 — a benchmark for how AI systems improve in online settings (most benchmarks assume models are stateless, but deployed systems should learn from experience). The task asks an agent to answer sequential questions against a SQL database; each question is a separate agent session, with memory provided via CMA memory (a mounted filesystem shared across sessions).

§ 05The memory progression

Effective use of memory follows a progression — and how far a model climbs it is what separates the three:

  1. Fail — get something wrong and document it.
  2. Investigate — before moving on, figure out why.
  3. Verify — turn the diagnosis into a checked fact.
  4. Distill — turn verification into a general rule.
  5. Consult — read the rule instead of re-deriving it.
ModelClimbs toBehavior
Sonnet 4.6Step 1 — FailStore is a list of failure notes and open guesses ("maybe prc instead of prc_usd?"); rarely consults prior notes.
Opus 4.7Step 3 — VerifyBuilds a schema reference with uncertainty flagged, but verification coverage is low: 7–33% of questions (median ~17%).
Fable 5Step 5 — ConsultCompletes the progression; strongest runs reach 73% verification coverage (22 of 30) and distill learnings into reusable rules.
The payoff

Fable 5 tends to complete the full progression — distilling general rules that carry forward and help with future tasks, rather than re-deriving the same facts each session.

§ 06Design loops, don't steer

Rather than directly prompting and steering Fable 5, it's often better to design loops that let the model self-correct in response to environment feedback (/goal or Outcomes) and manage its own context (via memory). These are just a few small-scale experiments — it's worth testing Fable 5 yourself on challenging tasks, using loops for self-correction or memory.
